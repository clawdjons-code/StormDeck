Product vision

StormDeck is a game-engine-powered 4D severe-weather cockpit that converts rapid PAR/NEXRAD radar volumes and radar-derived objects into a geospatially aligned, time-scrubbable storm workspace. For ATD v0, it should let an operator load an archived KATD case, inspect reflectivity/velocity/dual-pol fields in true sector geometry, slice horizontally/vertically, select a semantic storm object, and see what changed in roughly the last 60 seconds.

Operating principle: watch the storm evolve, not jump. PAR is valuable because it reduces mental interpolation between slow scans. Revision v0.2 adds explicit gates for the most likely failures: weak case choice, stale catalog links, false 360° coverage, bad transforms, QC artifacts, naive interpolation, GPU cliffs, and UX theater.

Recommended stack

  • Data exploration: Python, Py-ART, xarray/netCDF4, numpy, scipy.
  • Storage/interchange: raw CfRadial retained; render cache as Zarr or chunked binary/texture atlas; metadata as JSON.
  • Geospatial: pyproj for radar-to-earth coordinates; vector overlays from GeoJSON.
  • Renderer MVP: Unity for fastest workstation demo; preserve a WebGPU path for shareable browser views. Keep parser/cache schemas engine-neutral.
  • Streaming later: WebSocket/gRPC/NATS from a storm-state server.

Dream architecture

Live PAR/NEXRAD ingest at radar edge; edge QC/dealiasing/grid generation; central storm-state server; semantic storm-object tracking; multi-user forecaster cockpit; emergency-manager impact view; replay/training simulator; future adaptive PAR tasking requests. Adaptive scan/tasking stays a future capability, never a v1 dependency.

Latency budget target

  • Radar file/frame availability: 0–5 s after sweep/volume close for replay/live-adjacent source.
  • Edge parse/QC/normalization: 1–3 s per ATD volume for MVP workstation cache.
  • Derived deltas/object update: under 1 s for simple manual/heuristic objects.
  • Client transfer/cache load: 0.5–2 s for local replay, 2–5 s for networked demo.
  • Render response: 30–60 FPS desktop, interaction latency under 100 ms.

Data classes

  • Raw I/Q: radar-engineering future path; not v0 visualization input.
  • Moment/base data: v0 source of truth: reflectivity, velocity, spectrum width, ZDR, PHIDP, RHOHV; KDP optional.
  • Gridded/render-ready volume: derived cache, explicitly marked interpolated.
  • Derived products: rotation tracks, hail/debris signatures, trends; confidence-labeled.
  • Semantic storm objects: selectable storm/hook/mesocyclone/corridor entities; manual first, AI later.
ATD public data are minimally quality controlled research data. Show QC/provenance instead of hiding awkward artifacts. Radar data: famously honest, inconvenient, and petty.

Core data schemas

Use small JSON documents as stable contracts between parser, storm-state server, and renderer. Keep raw files immutable and put derived/interpolated status in metadata.



Reflectivity
Use conventional dBZ color ramp, but support opacity by confidence/QC. Clip extreme values with clear legend labels. Never smooth away hook-echo structure.
Velocity
Use diverging inbound/outbound palette with folded/aliased gates flagged. Couple velocity view with storm-relative option and rotation object overlays.
Dual-pol
ZDR, PHIDP, RHOHV appear as diagnostic overlays, not candy paint. Preserve calibration and steering-angle caveats.
Rotation
Derived shear/azimuthal-shear objects must be outlined as inference. Show time trend, confidence, and source fields used.
Hail/debris
Hail core/debris indicators require dual-pol logic and should be labeled probabilistic. Use textures/outlines rather than opaque blobs covering base radar.
Warning/impact zones
Warning polygons and corridors are decision overlays. Emphasize towns, roads, time-to-impact, and uncertainty cones over dramatic storm visuals.
Non-negotiable truth labels
  • Observed radial bin vs interpolated grid vs derived AI/object.
  • Scan start/end time, scan age, field name, units, radar ID, archive URL.
  • Coverage footprint of ATD sector; blank unsampled space stays blank.
  • Confidence/QC badge for minimal QC, clutter, range folding, velocity aliasing, ZDR bias, sidelobe artifacts.


Operator navigation model
  • Primary map: geospatial 3D scene centered on the ATD sector and storm area.
  • Timeline: scrub through volumes; pin before/after comparison at 60 s interval.
  • Slices: horizontal altitude slice, vertical cross-section drawn by user, native PPI sweep view.
  • Object inspector: selected storm object shows movement, intensity trend, fields used, confidence, and warning implications.
  • What changed panel: top deltas in reflectivity, velocity couplet, object motion, warning corridor impact.
  • Export: screenshot/video with timestamp/source/QC watermark baked in.
Modes
  • Forecaster mode: lowest cognitive load, warnings and deltas first.
  • Research mode: exposes fields, sweeps, calibration, QC, sampling geometry.
  • Training mode: replay controls, bookmarks, annotations, instructor prompts.
  • Emergency manager mode: simplified impact corridor, towns/assets, ETA and confidence.


First 2-week feasibility plan — revised build order
Days 1–2: Case scanner. Enumerate public ATD dates/files, verify URLs, and score candidate cases before choosing. Gate: case has usable reflectivity + velocity narrative
Days 3–4: CfRadial truth reporter. Emit per-file dimensions, variables, units, sweep metadata, time span, sector coverage, fill values, and QC caveats. Gate: no renderer until truth report passes
Days 5–6: Native PPI + geospatial validation. Render observed bins first; overlay radar site, range rings, towns, and ATD sector mask. Gate: blank unsampled space remains blank
Days 7–8: Timeline replay. Use actual frame timestamps and irregular intervals, not assumed cadence. Show scan start/end/duration/age.
Days 9–10: Slices with truth labels. Distinguish native sweep, interpolated slice, and derived grid. Expose sample density/observed mask.
Days 11–12: Manual semantic object + 60-second delta. Start conservative: manual object, evidence-linked deltas, actual comparison interval.
Days 13–14: Demo export. Bake timestamp, radar ID, field, source URL, observed/interpolated/derived status, and QC badge into screenshots/video.

Step-by-step failure modes and gates
StormDeck’s hard problem is not making radar look dramatic. It is making radar look dramatic without lying. Each build step gets a failure gate before downstream work can trust it.


Risk register
Misleading 3D interpolation
Pretty volume rendering can imply observations in unsampled space. This is the primary credibility hazard.
High
ATD research-data artifacts
Minimal QC, steering-angle polarimetric effects, clutter, sidelobes, velocity aliasing, and ZDR/phase bias must remain visible.
High
GPU memory blow-up
Multiple fields × time history × volume bricks can exceed workstation budgets.
Med
Geospatial trust failure
Subtle coordinate errors destroy credibility faster than any UI sin. Which is impressive.
Med
Live PAR access
Not required for v0 because replay mode uses public archive.
Low v0
